# Unit of Work Factory Pattern

## Overview

EfCore.Boost applications commonly use a **factory-based pattern** to create Unit of Work (UoW) instances on demand.

Instead of injecting a DbContext or UoW directly, services receive a factory that creates fresh UoW instances when needed.

---

## Why use a factory?

The factory pattern allows:

- Safe usage from singleton services  
- Multiple database connections  
- Explicit control over UoW lifetime  
- Clean separation of concerns  

---

## Lazy Creation (Why this matters)

The factory itself is lightweight and does **not** create a database context.

This ensures:

- No database resources are used until needed  
- Singleton services can safely depend on the factory  
- Each operation gets a fresh Unit of Work  

---

## Example Implementation

> ⚠️ Names like `BoostXUow` are generated by the template and will match your project name.

### Unit of Work

    public class BoostXUow(IConfiguration cfg, string connectionName)
        : UowFactory<BoostCTX>(cfg, connectionName)
    {
        public EfLongIdRepo<IpInfo> IpInfos => new(Ctx, DbType);
    }

### Factory

    public interface IUowBoostXFactory
    {
        BoostXUow Create(string? connectionName = null);
    }

    public sealed class UowBoostXFactory(IConfiguration cfg) : IUowBoostXFactory
    {
        public BoostXUow Create(string? connectionName = null)
        {
            var resolvedName = string.IsNullOrWhiteSpace(connectionName)
                ? cfg["DefaultAppConnName"] ?? throw new InvalidOperationException("Missing 'DefaultAppConnName'")
                : connectionName;

            return new BoostXUow(cfg, resolvedName);
        }
    }

---

## Registering the Factory

    builder.Services.AddSingleton<IUowBoostXFactory, UowBoostXFactory>();

---

## When NOT to use the factory

In console tools or migration programs, direct construction is simpler:
```csharp
    using var uow = new BoostXUow(configuration, connectionName);
```

---

## Best Practices

- Create a new UoW per operation  
- Dispose immediately (`using var`)  
- Inject the factory, not the UoW  
- Keep the factory lightweight  

---

## Summary

The UoW factory pattern:

- Centralizes creation  
- Delays database usage until needed  
- Works cleanly with DI  
- Keeps application logic simple  

Use the factory in application code.  
Use direct construction in migration tools.
